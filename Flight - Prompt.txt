## 3.2 Database Schema Overview

```sql
-- USERS
CREATE TABLE users (
    id          UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    full_name   VARCHAR(100) NOT NULL,
    email       VARCHAR(150) UNIQUE NOT NULL,
    mobile      VARCHAR(15),
    password_hash TEXT NOT NULL,
    is_guest    BOOLEAN DEFAULT FALSE,
    created_at  TIMESTAMPTZ DEFAULT NOW()
);

-- GUEST LOGIN TRACKER
CREATE TABLE guest_login_attempts (
    id          SERIAL PRIMARY KEY,
    identifier  VARCHAR(200) NOT NULL,  -- IP or fingerprint
    attempt_count INT DEFAULT 0,
    last_attempt TIMESTAMPTZ DEFAULT NOW(),
    UNIQUE(identifier)
);

-- CITIES (Static - pre-seeded)
CREATE TABLE cities (
    id          SERIAL PRIMARY KEY,
    name        VARCHAR(100) NOT NULL,
    iata_code   VARCHAR(3) UNIQUE NOT NULL,
    country     VARCHAR(100),
    timezone    VARCHAR(50)
);

-- AIRLINES (Static - pre-seeded)
CREATE TABLE airlines (
    id          SERIAL PRIMARY KEY,
    name        VARCHAR(100) NOT NULL,
    iata_code   VARCHAR(2) UNIQUE NOT NULL,
    logo_url    TEXT
);

-- FLIGHTS (Static with recurring schedule)
CREATE TABLE flights (
    id              UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    airline_id      INT REFERENCES airlines(id),
    flight_number   VARCHAR(10) NOT NULL,
    origin_id       INT REFERENCES cities(id),
    destination_id  INT REFERENCES cities(id),
    departure_time  TIME NOT NULL,
    arrival_time    TIME NOT NULL,
    duration_mins   INT NOT NULL,
    base_price_adult    DECIMAL(10,2),
    base_price_child    DECIMAL(10,2),
    base_price_newborn  DECIMAL(10,2),
    available_days  VARCHAR(20),  -- e.g. "MON,WED,FRI"
    total_seats     INT DEFAULT 180,
    available_seats INT DEFAULT 180
);

-- BOOKINGS
CREATE TABLE bookings (
    id              UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    booking_ref     VARCHAR(10) UNIQUE NOT NULL,
    user_id         UUID REFERENCES users(id),
    flight_id       UUID REFERENCES flights(id),
    return_flight_id UUID REFERENCES flights(id),
    travel_date     DATE NOT NULL,
    return_date     DATE,
    trip_type       VARCHAR(10) CHECK (trip_type IN ('ONE_WAY', 'RETURN')),
    total_amount    DECIMAL(10,2),
    status          VARCHAR(20) DEFAULT 'PENDING',
    created_at      TIMESTAMPTZ DEFAULT NOW()
);

-- PASSENGERS
CREATE TABLE passengers (
    id          UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    booking_id  UUID REFERENCES bookings(id),
    full_name   VARCHAR(100) NOT NULL,
    age_category VARCHAR(10) CHECK (age_category IN ('ADULT','CHILD','NEWBORN')),
    seat_number VARCHAR(5)
);

-- ADD-ONS SELECTED
CREATE TABLE booking_addons (
    id          SERIAL PRIMARY KEY,
    booking_id  UUID REFERENCES bookings(id),
    addon_type  VARCHAR(50),  -- BAGGAGE, MEAL, INSURANCE, SEAT
    description TEXT,
    quantity    INT DEFAULT 1,
    unit_price  DECIMAL(10,2),
    total_price DECIMAL(10,2)
);

-- PAYMENTS
CREATE TABLE payments (
    id              UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    booking_id      UUID REFERENCES bookings(id),
    payment_method  VARCHAR(20) CHECK (payment_method IN ('CREDIT_CARD','DEBIT_CARD','PAYPAL','UPI')),
    amount          DECIMAL(10,2),
    transaction_ref VARCHAR(50),
    status          VARCHAR(20) DEFAULT 'PENDING',
    initiated_at    TIMESTAMPTZ DEFAULT NOW(),
    confirmed_at    TIMESTAMPTZ
);
```

# SECTION 4 — AUTHENTICATION, SESSION & REGISTRATION
# ══════════════════════════════════════════════════════════════════════

## 4.1 Login Page — Complete Specification

### Visual Design
- Centered card (max-width 440px) on a gradient background (navy→sky blue)
- Airline brand logo at the top of the card
- Subtle welcome text: "Welcome Back. Your journey continues here."
- Username and Password fields with floating labels
- Password field has a show/hide toggle (eye icon)
- "Remember Me" checkbox (stores username only — never password)
- Primary CTA button: "Sign In" — full width, rounded corners
- Secondary link: "Don't have an account? Register here"
- Guest info banner (dismissible): "Exploring? Use admin / admin@123 — 20 free sessions available"

### Default Credentials Display
- Show a collapsible info box on the login page:
  ```
  🔑 Guest Access
  Username: admin
  Password: admin@123
  Valid for 20 sessions. Register for unlimited access.
  ```
- The info box should have a "Use These Credentials" button that auto-fills the fields
- Counter displayed: "Guest sessions remaining: X / 20"

### Guest Login Limit Logic
- Track attempts via: IP address + browser fingerprint combination stored in `guest_login_attempts` table
- Counter increments on every successful login with default credentials
- At exactly attempt 21 and beyond:
  - Block the login attempt
  - Show modal: "You've used all 20 guest sessions. Please create a free account to continue booking."
  - Modal has two buttons: "Register Now" and "Cancel"
  - The "Sign In" button is disabled and shows tooltip: "Guest limit reached. Please register."
- Counter is NOT reset by clearing cookies — it is server-side persistent
- Admin credentials (when logged in as actual admin) bypass the guest counter

### Form Validations — Login
| Field    | Validation Rule                                            | Error Message                          |
|----------|------------------------------------------------------------|----------------------------------------|
| Username | Required, min 3 chars, no special chars except @ _ .      | "Username is required"                 |
| Password | Required, min 6 chars                                      | "Password is required"                 |
| Both     | Credentials must match a user record                       | "Invalid username or password"         |
| Rate Limit| Max 5 failed attempts in 10 minutes per IP                | "Too many attempts. Try again in X min"|

### Error Handling — Login
- Invalid credentials: Show inline error below the password field (not above the form)
- Account locked: Show specific message with unlock instructions
- Network error: Show toast at top: "Unable to connect. Please check your connection."
- Server error: "Login service is temporarily unavailable. Please try again."
- On success: Show brief success toast → redirect to home (200ms delay for smoothness)

## 4.2 Registration Page — Complete Specification

### Fields Required
| Field            | Type     | Validation                                              |
|------------------|----------|---------------------------------------------------------|
| Full Name        | Text     | Required, 2–100 chars, letters/spaces/hyphens only      |
| Email Address    | Email    | Required, valid format, unique in system                |
| Mobile Number    | Tel      | Required, 10 digits (India) or E.164 international      |
| Password         | Password | Required, min 8 chars, 1 uppercase, 1 digit, 1 special  |
| Confirm Password | Password | Required, must exactly match Password field             |
| Terms Checkbox   | Checkbox | Must be checked to submit                               |

### Real-Time Validation Behavior
- Validate on `blur` (when field loses focus) — not on every keystroke
- Show green checkmark (✓) when field passes validation
- Show red error text below field when validation fails
- Password strength meter (Weak / Fair / Strong / Very Strong) shown while typing
- "Confirm Password" field checks match on every keystroke after first character

### Error Handling — Registration
- Duplicate email: "An account with this email already exists. Try logging in instead."
- Password mismatch: "Passwords do not match."
- Mobile format error: "Please enter a valid mobile number."
- Server error during registration: "Registration failed. Please try again in a moment."
- On success: Auto-login the user → redirect to home with welcome toast

## 4.3 Session Management — Complete Specification

### Session Timeout Rules
- Session duration: exactly 60 minutes from login time
- A countdown timer is visible in the sticky header: "Session: 45:22 remaining"
- At 5 minutes remaining: Yellow warning banner appears: "Your session expires in 5 minutes. Click here to extend."
- "Extend Session" refreshes the JWT via the `/api/auth/refresh` endpoint without full re-login
- At 0 minutes: All API calls return 401 → automatic redirect to login page
- Message on login page: "Your session has expired. Your booking progress has been saved."
- Server-side token invalidation on expiry — client-side clearing alone is NOT sufficient

### Logout Behavior
- Logout button in header: always visible
- On click: Confirmation modal: "Are you sure you want to log out? Any unsaved booking will be lost."
- Modal buttons: "Yes, Log Out" (red) and "Stay Logged In" (outlined)
- On confirm: Call `/api/auth/logout`, invalidate token server-side, clear client state, redirect to login
- Toast on login page after logout: "You have been logged out successfully."

## 5.1 Page Layout & Visual Design

- Page title: "Where would you like to go?" with a subtle plane icon
- Trip type toggle at the top (pill-style buttons): "✈ One Way" | "⇄ Return"

- "Search Flights" button is large, primary-colored, full-width at the bottom
- Modify Search mode pre-fills all fields from the last search for quick editing

## 5.2 Form Fields — Complete Specification

### Origin & Destination (Autocomplete Dropdowns)
- Typing triggers a debounced (300ms) search against the cities list
- Display format in dropdown: "New Delhi (DEL) — India"
- Keyboard navigable (arrow keys + Enter to select)
- If no matching city: Show "No cities found. Try a different name."
- Recently searched cities shown on focus (stored in localStorage, max 5)
- Origin and Destination cannot be the same city (validated on blur and on submit)

### Travel Date / Return Date
- Date picker opens as a calendar overlay
- All past dates are visually greyed out and unclickable
- Today's date is selectable only if there are flights available later today
- For Return trips: The return date picker opens only after departure date is selected
- Return date calendar highlights the departure date and disables all dates before it
- Date format displayed: "Mon, 15 Jan 2025"
- On mobile: Native date picker used for better UX

### Passenger Counter
- Displayed as an expandable panel: "1 Adult  ▼"
- On expand, shows three rows with +/- buttons:
  - Adults (12+ years): Min 1, Max 9
  - Children (2–11 years): Min 0, Max 8
  - Newborns (0–23 months): Min 0, Max 4
- Total passenger count shown: "Total: 3 passengers"
- The minus (-) button is disabled when count is at its minimum
- The plus (+) button is disabled when total passengers reach 9
- ARIA labels: "Increase adult count", "Decrease adult count" etc.

## 5.3 Business Rules & Guard Conditions

| Rule                                              | Error Message (exact)                                         |
|---------------------------------------------------|---------------------------------------------------------------|
| At least 1 adult required to search               | "Please add at least 1 adult passenger to search flights."   |
| Newborn without adult is not allowed              | "A newborn must be accompanied by at least 1 adult."         |
| Origin must be selected                           | "Please select your departure city."                         |
| Destination must be selected                      | "Please select your arrival city."                           |
| Origin and Destination must differ                | "Departure and arrival cities cannot be the same."           |
| Travel date must be selected                      | "Please select your travel date."                            |
| Travel date must be in the future                 | "Travel date must be today or in the future."                |
| Return date required for Return trip              | "Please select your return date."                            |
| Return date must be >= travel date                | "Return date must be the same as or after your travel date." |
| Total passengers cannot exceed 9                  | "Maximum 9 passengers allowed per booking."                  |

## 5.4 Search Loading State
- On "Search Flights" click:
  1. Button text changes to "Searching..." with a spinner icon
  2. Button is disabled to prevent double-submit
  3. Skeleton loader appears on the results area (shimmer effect)
  4. If results take >3 seconds: Show "Still searching, just a moment..."
  5. If no results found: Show friendly empty state (illustration + message)
  6. On error: Show error toast with Retry button

### Card Number Validation — Technical Detail
- Strip spaces before validation

## 10.5 Payment Submission Flow

### Step 1: Client-Side Validation
- All fields validated inline before submission
- First failing field is auto-focused with error message
- Submit button shows "Paying..." with spinner during processing

### Step 2: Payment Initiation API Call
- POST `/api/payment/initiate` with booking ID + payment method + amount
- On success: Receive a `payment_token` and `redirect_url`
- On failure: "Payment initiation failed. Please check your details and try again."

### Step 3: Dummy Bank Redirect
- Open the `redirect_url` (could be same tab redirect or iframe/modal)
- Dummy bank page shows:
  - Bank logo and "Processing Payment" heading
  - Animated progress bar that fills over 4–7 seconds
  - Status text transitions: "Verifying details..." → "Contacting bank..." → "Authorizing..." → "Complete!"
  - Then auto-redirects back to the booking app with a `?status=SUCCESS&token=XXX` parameter

### Step 4: Callback Handling
- App receives the callback, extracts token and status
- POST `/api/payment/callback` with the token
- On SUCCESS:
  - Update booking status to CONFIRMED in DB
  - Generate booking reference ID (format: FLT-XXXXXXXX)
  - Trigger confirmation email (mock)
  - Redirect to Confirmation Page
- On FAILURE:
  - Return to payment page with error: "Payment was unsuccessful. Please try a different payment method or card."
  - Booking status remains PENDING — not charged again

### Step 5: Security Checks
- Validate the callback token server-side — never trust client-passed status
- Token is single-use — a replayed token must return an error
- Amount in callback must match the original booking amount
- All payment API calls must use HTTPS only

## 10.6 Error Handling — Payment
| Scenario                          | User-Facing Message                                           |
|-----------------------------------|---------------------------------------------------------------|
| Invalid card number               | "Enter a valid 16-digit card number"                         |
| Expired card                      | "This card has expired. Use a different card."               |
| Insufficient funds (simulated)    | "Payment declined. Check your balance or try another method."|
| Network timeout during redirect   | "Payment is taking longer than expected. Do not refresh."    |
| Callback token tampered           | "Payment verification failed. Contact support."              |
| Duplicate payment attempt         | "This booking has already been paid."                        |
| Server error                      | "Payment service error. Your card has NOT been charged."     |

```
POST   /api/auth/login
```

```
GET    /api/cities                   → [ { id, name, iata, country } ]

```

```
POST   /api/booking/create           → creates PENDING booking
PUT    /api/booking/:id/passengers   → saves passenger details
PUT    /api/booking/:id/addons       → saves selected add-ons
GET    /api/booking/:id/summary      → full booking summary
POST   /api/booking/:id/confirm      → confirms booking post-payment
GET    /api/booking/:id/ticket       → returns ticket data for display/PDF
```

## 13.4 Payment APIs

```
POST   /api/payment/initiate         → returns { payment_token, redirect_url }
POST   /api/payment/callback         → processes bank return { token, status }
GET    /api/payment/:bookingId/status → payment status check
```

## 13.5 Utility APIs

```
POST   /api/email/resend-confirmation
GET    /api/ticket/:bookingRef/pdf   → returns PDF file
```

## 13.6 Standard API Response Format

```json
// Success
{
  "success": true,
  "data": { ... },
  "message": "Operation successful",
  "timestamp": "2025-01-15T10:30:00Z"
}

// Error
{
  "success": false,
  "error": {
    "code": "VALIDATION_ERROR",
    "message": "Card number must be 16 digits",
    "field": "card_number"
  },
  "timestamp": "2025-01-15T10:30:00Z"
}
```

## 13.7 HTTP Status Codes Used

| Code | Meaning               | Usage                                    |
|------|-----------------------|------------------------------------------|
| 200  | OK                    | Successful GET / PUT                     |
| 201  | Created               | Successful POST (new booking/user)       |

| 403  | Forbidden             | Action not permitted (e.g. re-access payment) |

| 409  | Conflict              | Duplicate email / duplicate payment      |

# SECTION 15 — GLOBAL ERROR HANDLING REFERENCE
# ══════════════════════════════════════════════════════════════════════

## 15.1 Frontend Error Handling Rules
- All API calls use async/await inside try/catch blocks
- Loading state is always set to TRUE before call, FALSE in finally block
- Submit buttons disabled during loading (prevents double-submit)
- Network errors show toast with Retry button
- 401 responses silently redirect to login
- 422/400 validation errors map field names to inline form errors
- 5xx errors show: "Something went wrong. Our team has been notified."
- Error boundaries wrap all major page components (React Error Boundary)
- All errors logged to console with: endpoint, payload, response, timestamp

## 15.2 Backend Error Handling Rules
- All routes wrapped in global error middleware
- Validation layer (Joi/Zod) runs before business logic
- Database errors never exposed to client (log internally, return generic 500)
- Auth failures always return 401 with generic message (never reveal why)
- All incoming data sanitized and validated for type, length, and format
- Rate limiting applied on: login, register, payment initiate endpoints
- CORS configured to allow only the frontend domain

## 15.3 Form Error Display Rules
- Errors appear directly below the failing field (not in a banner above)
- Error text is red (#D32F2F) with a ✕ icon prefix
- Error clears automatically when user starts correcting the field (on input)
- On form submit failure, page auto-scrolls to the first error field
- Field border turns red on error, green on valid, default grey on untouched
- All error messages use plain language — no technical jargon

## 15.4 Loading & Async State Rules
- Every button that triggers an API call shows a spinner and changes text
- Skeleton loaders for all data-driven lists (flight results, booking summary)
- Full-page loader for heavy operations (payment redirect)
- "Still loading..." secondary text appears if any operation exceeds 3 seconds
- All loading states are accessible (aria-busy="true" on loading containers)

# SECTION 17 — AGENT OPERATIONAL PRINCIPLES
# ══════════════════════════════════════════════════════════════════════

1.  **Requirement Traceability**
    Every test case maps to a section in this spec. Never create orphan tests.

2.  **Test Data Independence**
    Use parameterized, synthetic data. Never hardcode production values.

3.  **Error Message Exactness**
    Verify the exact error text, not just that an error appears.

4.  **State Preservation**
    Back navigation must always preserve user-entered data across all pages.

5.  **Security Testing Mindset**
    Every module must have ≥1 security test: SQL injection, token tampering,
    unauthorized URL access, XSS in input fields.

6.  **Accessibility Compliance**
    Tab navigation, ARIA labels, and screen reader compatibility are
    first-class test requirements — not optional.

7.  **Cross-Browser & Responsive Testing**
    Execute critical paths on Chrome, Firefox, Safari, Edge.
    Test at 375px (mobile), 768px (tablet), 1440px (desktop).

8.  **Performance Baseline**
    - Search results must render within 3 seconds under normal load
    - Page transitions must feel instant (<200ms perceived delay)
    - Payment redirect must not exceed 10 seconds total

9.  **Regression Discipline**
    After every bug fix, re-run all test cases in the affected module
    plus all integration points with that module.

10. **Double-Submit Prevention**
    Verify that every form submission button is disabled during loading
    to prevent duplicate bookings, duplicate payments, or duplicate registrations.

11. **Session State Integrity**
    Verify booking data is preserved across session extension. Verify it
    is cleared after logout but can be recovered with "saved booking" if
    session expires unexpectedly.

12. **Smooth UX Validation**
    Test that all transitions, animations, and loading states render
    smoothly without flickering, layout shifts, or invisible elements.
